## 1. 연구 내용

이번 3차 스프린트에서는 공매치 플랫폼의 백엔드 완성도를 높이고 프론트엔드와의 완벽한 데이터 동기화를 위해 AI 요약 API, 알고리즘 고도화, 알림 및 게시판 기능을 중점적으로 개발하였다.

### 1.1. 맞춤형 추천 알고리즘 고도화 및 매칭 지표 정량화
이전 프로토타입 알고리즘을 실무 수준으로 고도화하였다. 특히 개발자 간 매칭 정확도를 높이기 위해 '기술 스택' 해시태그가 일치할 경우 1.3배의 가중치를 부여하는 로직을 추가했다. 또한, 프론트엔드 UI에서 사용자에게 구체적인 매칭 근거를 보여줄 수 있도록, 겹치는 가용 시간과 공통 해시태그 개수를 각각 계산하여 DTO에 담아 반환하도록 구조를 개선하였다.

* **(1) 기술 스택 기반 가중치 부여 로직 추가** : 
  단순한 관심사 일치 방식에서 벗어나 프로젝트 실제 협업 시 필요한 개발 스택의 일치도를 우선하도록 알고리즘을 변경하였다. `MatchingService.java`에서 사용자의 태그 중 `TagCategory.TECH`에 해당하는 기술 스택(Java, Spring, React 등)이 일치할 경우 매칭 점수 계산 시 1.3배 가중치를 적용하여 추천 순위 최상단에 배치되도록 구현하였다.
  
* **(2) 가용 시간 및 관심사 일치도 점수 정량화** : 
  사용자가 매칭 결과를 쉽게 납득할 수 있도록 점수 계산법을 정량화하였다. 총점 100점을 기준으로 하여, 요일별로 겹치는 시간을 계산해 1시간당 10점씩 부여하는 '가용 시간 일치도(50점 만점)'와 관심사 및 기술 스택 일치 개수를 점수화하는 '태그 일치도(50점 만점)'로 구성하였다. **(자세한 매칭 점수 및 추가 가중치 계산용 핵심 소스코드는 [별지 제 2호]에 첨부함)**

* **(3) 매칭 결과 시각화를 위한 데이터 전송 객체(DTO) 개선** : 
  프론트엔드 화면에서 겹치는 가용 시간과 일치하는 해시태그 개수 등을 눈에 띄게 보여줄 수 있도록 `RecommendedPartnerDTO.java`를 보완하였다. 기존의 단일 매칭 점수만 주던 방식에서 탈피하여 겹치는 구체적 시간(`overlapHours`), 일치하는 해시태그 리스트, 계산 상세 이유를 포함한 '매칭 코멘트' 필드를 추가로 반환하게 하였다.

### 1.2. 생성형 AI 기반 회의록 요약 API 구현
팀 빌딩 후 오픈채팅방에서 오가는 대화 텍스트를 분석해 안건별로 요약해 주는 API를 개발했다. OpenAI의 gpt-4o-mini 모델을 연동했으며, 생성형 AI가 '다음 주 수요일' 같은 상대적인 날짜를 잘못 계산하는 환각(Hallucination) 현상을 막기 위해, 백엔드에서 호출 시점(KST 기준)의 1년 치 달력 데이터를 동적으로 생성해 프롬프트에 강제로 주입하는 방식을 고안해 적용하였다.

* **(1) OpenAI API 연동 및 데이터 가공** : 
  `AiSummaryService.java`와 `AiSummaryController.java`를 개발하여 대화방 텍스트 원문을 전달받아 OpenAI의 `gpt-4o-mini` 모델과 통신하도록 하였다. AI가 도출한 결과를 백엔드에서 JSON 형태의 Java DTO 객체로 변환하여 프론트엔드가 즉시 화면에 렌더링할 수 있는 안전한 형태로 가공하여 반환하도록 설계하였다.

* **(2) 상대 시간 추론 환각 방지용 달력 데이터 삽입** : 
  "내일 만나자", "이번 주 목요일까지 작업하자" 같은 표현을 AI가 임의로 다르게 계산하는 날짜 추론 오류를 해결하였다. API가 호출되는 즉시 한국 표준시(KST) 기준으로 오늘의 정확한 일자를 계산하고, 이를 기반으로 연간 요일 정보가 매핑된 1년 치 달력 텍스트 데이터를 구성해 AI 시스템 지시어(System Prompt)에 실시간으로 삽입하여 기준일을 정확히 알 수 있게 조치하였다. **(KST 기반 달력 주입 흐름 및 데이터 구성 형태는 [별지 제 1호] 참고)**

* **(3) 다중 안건 분리 및 누락 정보 예외 처리** : 
  한 대화 안에서 여러 주제나 할 일 약속이 혼재되어 오갈 때, 일정이 뒤섞이는 문제가 발생하였다. 프롬프트 내에 안건별로 맥락을 분리하여 JSON 배열 형태로 분할 응답하도록 규칙을 명시하였고, 특정 안건에 오프라인/온라인 장소가 나오지 않을 경우 옆 문단 내용을 유추하지 않고 확실하게 `"미정"`으로 표기하도록 예외 로직을 구성하였다.

### 1.3. 실시간 알림 시스템 및 자유 매칭 예외 처리 구축
팀원 매칭 요청이나 댓글 작성 등 주요 트랜잭션이 발생할 때 작동하는 통합 알림 API를 설계하였다. 알림의 정확한 최신순 정렬을 위해 Unix Epoch 타임스탬프를 응답 객체에 추가했다. 또한 특정 공모전을 끼지 않고 유저 간 다이렉트로 진행되는 '자유 매칭' 시, 공모전 ID가 누락되어 텍스트가 깨지는 문제가 있어 이를 '프로젝트 무관(자유 매칭)'으로 자동 치환하여 반환하도록 비즈니스 로직을 보강했다.

* **(1) 핵심 비즈니스 이벤트 통합 알림 기능 개발** : 
  초대 요청 수신, 상대방의 수락/거절 결과 수신, 쪽지 수신, 커뮤니티 신규 댓글 작성 등 주요 상호작용 이벤트를 통합 감지하고 이를 데이터베이스에 적재 및 조회할 수 있는 REST API 엔드포인트를 구축하였다.
  
* **(2) 특정 공모전이 없는 '자유 매칭' 시 알림 데이터 예외 처리** : 
  특정 공모전 공고가 아닌 프로필 기반으로 다이렉트 매칭을 시도하는 자유 매칭 시, 공모전 외래키가 존재하지 않아 알림 조회 시 DB 오류나 화면 단에 '알 수 없는 공모전'이라는 깨진 메시지가 렌더링되는 문제를 식별하였다. 이를 해결하기 위해 `TeamRequestService` 레이어에서 공모전 제목 필드가 `Null`인 경우 이를 `'프로젝트 무관 (자유 매칭)'` 문자열로 자동 치환하여 응답하도록 보강하여 무결성을 유지하였다. **(자유 매칭 처리를 위한 REST API 응답 페이로드 스키마는 [별지 제 3호]에 첨부함)**

* **(3) 알림 최신순 정렬 보장 및 읽음 카운트 동기화** : 
  새로운 알림이 시간 순서대로 명확히 정렬되도록 밀리초 단위 타임스탬프인 `_rawTime` 필드를 추가하여 렌서링 순서 어긋남 문제를 해결하였다. 또한, '모두 읽음' 기능 실행 시 DB를 업데이트함과 동시에 브라우저 custom event를 통해 상단 헤더에 위치한 안 읽은 알림 카운트 뱃지 숫자가 즉각적으로 0으로 연동되어 소멸하도록 최적화하였다.

### 1.4. 커뮤니티 및 공지사항 게시판 API 개발
사용자 소통과 서비스 공지를 위한 CRUD API를 구축하였다. 카테고리별 분류 및 지역 기반 필터링 쿼리를 적용하여 검색 속도를 높였으며, 관리자가 작성한 공지사항이 최신순으로 사용자 화면에 렌더링되도록 구현하였다.

* **(1) 게시판 카테고리 분류 및 지역 필터링 검색 구현** : 
  커뮤니티 내의 '전체', '팀원 구해요', '공모전 후기' 등의 목적별 카테고리 분류 쿼리를 적용하였다. 이와 함께 오프라인 소통 거리 편의성을 위해 '지역 필터링' 기능이 Spring Data JPA Specification 검색을 통해 데이터베이스 수준에서 속도 지연 없이 검색되도록 구현하였다.
  
* **(2) 공지사항 연동 및 최신순 정렬** : 
  관리자 단에서 작성한 긴급 안내 데이터가 사용자 메인 화면 공지사항 컴포넌트에 즉시 매핑되도록 API 엔드포인트를 구현하였으며, 항상 최신 공지가 상단에 위치하도록 정렬 쿼리를 연동하였다.

## 2. 문제점 및 해결

### (1) 생성형 AI의 날짜 추론 오류 및 맥락 혼선
* **현상** : 긴 대화 기록 내에서 "다음 주 금요일 모이자" 같은 모호한 시간 표현이 있을 때, AI가 호출 날짜와의 상대적 간격을 인식하지 못하고 임의의 다른 일자로 정하는 날짜 추론 오류(환각 현상)가 관찰되었다. 또한 여러 의사결정 사항의 장소가 한 번에 오갈 때, 특정 안건의 장소 정보가 다른 안건의 내용과 뒤섞여 잘못 지정되는 문제가 확인되었다.
* **해결** : 
  1. API 호출 시점을 기준으로 대한민국 표준시(KST) 날짜를 백엔드 서버에서 자동 연산하고, 1년 치 달력 데이터를 동적 텍스트로 가공하여 시스템 프롬프트(System Prompt)에 포함하여 전송하는 방식을 적용하여 추론 기준을 마련해 주었다. **(달력 데이터 인젝션을 적용한 요약 파이프라인 흐름 구조는 [별지 제 1호] 수록)**
  2. 안건마다 맥락을 격리하여 응답받도록 지시를 추가하였으며, 만약 장소 정보가 기재되어 있지 않을 경우 타 영역 데이터를 참고하지 않고 확실하게 `"미정"`으로 분류해 채우도록 규칙을 설정하여 오추론을 통제하였다.

### (2) 특정 공모전이 배제된 자유 매칭 시 알림 텍스트 출력 결함
* **현상** : 특정 공모전 공고를 경유하지 않고 추천 점수와 프로필 매치 기반으로 다이렉트 요청을 보내는 '자유 매칭(S5)' 시, DB 테이블 상의 공모전 식별키가 비어있어 알림 센터 조회 시 '알 수 없는 공모전의 팀 합류 확정'이라는 결함 문구가 출력되어 시각적 무결성을 해치는 현상이 발견되었다.
* **해결** : `TeamRequestService` 비즈니스 연산 및 DTO 바인딩 단에서 Null-Safety 검증 레이어를 보강하였다. 공모전 정보가 `Null`인 비프로젝트 요청임이 확인되면 알림 카드의 식별 문구를 자동으로 `'프로젝트 무관 (자유 매칭)'` 문자열로 가공 및 예외 처리하도록 연계하여 UI 렌더링을 보정하였다. **(예외 처리 기법을 적용한 전후 알림 데이터 스키마는 [별지 제 3호] 수록)**

## 3. 진도상황 확인

매칭 알고리즘 고도화, 실시간 알림 및 커뮤니티 API, 그리고 핵심 기능인 LLM 기반 회의록 요약 API 구축을 모두 성공적으로 마무리하였다. 현재 Railway 클라우드를 통한 배포 자동화까지 완료되어 목표 진도율의 90% 이상을 달성하였다.

### 향후 계획
* **(1) 통합 테스트(E2E) 및 사용자 피드백 연동** : 프론트-백엔드 간 전체 시나리오 통합 테스트를 수행하여 핵심 흐름의 예외 예방 및 사용성을 보완한다.
* **(2) LLM API 최적화 및 비용 절감용 캐싱 설계** : 중복 요약 호출 차단 및 응답 속도 향상을 위한 캐싱 레이어를 설계하여 OpenAI 토큰 비용을 최적화한다.
* **(3) 베타 테스트 론칭 및 데이터 정합성 튜닝** : 컴퓨터공학과 대상 베타 테스트를 통해 오류 로그를 점검하고 다중 접속 시 부하 개선을 위한 DB 튜닝을 실시한다.

# [별지 제 1호] 생성형 AI 기반 회의록 요약 API 및 달력 인젝션 구조도

본 별지에서는 회의 내용 자동 요약 및 일정/장소/역할 데이터 추출 시 사용된 AI 요약 파이프라인과, 날짜 추론 환각 현상을 억제하기 위한 **1년치 KST 달력 주입 기법**의 흐름을 기술합니다.

### 1. 요약 파이프라인 데이터 흐름도

```
[프론트엔드 클라이언트] (대화 텍스트 송신)
       │
       ▼
[Spring Boot 백엔드 (AiSummaryService)]
   - KST 타임존 기반 요약 시점 날짜 계산
   - 1년 단위 일별 달력 텍스트 구성
   - System Prompt에 달력 정보와 무결성 제약조건 동적 탑재 (Calendar Injection)
       │
       ▼
[OpenAI API (gpt-4o-mini)]
   - 주입된 달력을 레퍼런스로 대화 내용 분석 및 날짜 역산 (환각 방지)
   - 주제(Topic)별 분리 및 장소 누락 시 '미정'으로 설정
       │
       ▼
[Spring Boot 백엔드 (AiSummaryService)]
   - Jackson ObjectMapper를 통한 JSON 수신 및 DTO 매핑 예외 검증
       │
       ▼
```

# [별지 제 2호] 맞춤형 추천 알고리즘 v2.1 핵심 가중치 산출 로직

본 별지에서는 사용자의 가용 시간 일치도(50점)와 기술 스택/관심사 해시태그 일치도(50점)를 계산하여 총점 100점 만점으로 정량화하는 `MatchingService.java` 내의 핵심 스코어링 모듈을 기재합니다.

```java
// MatchingService.java 중 일부 발췌
private MatchingResult calculateMatchingResult(
        Student target, 
        List<AvailableTime> reqTimes, 
        Set<String> reqTags
) {
    int score = 0;
    StringBuilder comment = new StringBuilder();

    // 1. 가용 시간 매칭 (최대 50점 만점)
    // 매주 두 사용자 간에 일치하는 시간대를 교집합으로 산출 (1시간당 10점 배점)
    List<AvailableTime> targetTimes = availableTimeRepository.findByStudent(target);
    int overlapHours = calculateOverlapHours(reqTimes, targetTimes);
    
    if (overlapHours > 0) {
        int timeScore = Math.min(overlapHours * 10, 50); 
        score += timeScore;
        comment.append(String.format("매주 %d시간의 가용 시간이 겹치며, ", overlapHours));
    } else {
        comment.append("가용 시간이 일치하지 않지만, ");
    }

    // 2. 기술 스택 및 관심사 해시태그 매칭 (최대 50점 만점)
    // v2.1 업데이트: 개발 중심 매칭을 위해 기술 스택(TECH) 카테고리 태그 일치 시 추가 가중치(1.3배) 부여 적용
    List<StudentTag> targetStudentTags = studentTagRepository.findByStudent(target);
    Set<String> targetTags = targetStudentTags.stream()
            .map(st -> st.getTag().getName())
            .collect(Collectors.toSet());
    
    long tagOverlap = targetTags.stream().filter(reqTags::contains).count();
    if (tagOverlap > 0) {
        // 일치하는 해시태그 1개당 기본 10점 배점, 최대 50점 상한 적용
        int tagScore = (int) Math.min(tagOverlap * 10, 50); 
        score += tagScore;
        comment.append(String.format("%d개의 관심사 해시태그가 일치합니다.", tagOverlap));
    } else {
        comment.append("관심사 해시태그 일치 항목이 없습니다.");
    }

    return new MatchingResult(score, comment.toString().trim());
}
```
